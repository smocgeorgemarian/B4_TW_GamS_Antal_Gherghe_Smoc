1. Introduction

1.1 Purpose

The following SRS describes an HTML/CSS/JavaScript architecture used for creating the FrontEnd side of the project "GamS (Gamification as a Service)". The current iteration has release number 1.3.1.

1.2 Document Conventions

The report is written using the font "Times New Roman". The statements and requirements that will be described have their own priority given by the "<h[number]>" node given by the "Scholarly HTML" format.

1.3 Intended Audience and Reading Suggestions

The document is intended for developers and project managers. The rest of the SRS report contains details about the given project and is organized in the "Scholarly HTML" format. The suggested sequence for reading the document is beginning with the "1. Introduction" section and proceeding through the sections that are most pertinent for each reader.

1.4 Product Scope

The software specified is a gamification service used by web developers and site owners to add rewards granted to users for doing specified tasks or actions and add a leveling system. The benefits of this service are on a marketing level to motivate and engage users. The main objective is to provide means for enhancing systems, services, and activities to create similar experiences to those experienced when playing games. Our goals are to create a reliable and easy way to achieve the objective described above. The FrontEnd side of the software is a replacement for other competitors like "Reddit", "Twitter", etc. which takes the best of each and creates a place where everybody can experience a new and exciting approach. This combined with the gamification service will let users 'choose' their own way of using our product.

2. Overall Description

2.1 Product Perspective

The product is a follow-on member of the social apps family. The main scope is to replace already existing apps by creating a new experience using the best things its competitors offer.

2.2 Product Functions

  1. Gamification System
    1. Leveling System
    2. Rewards System
  2. User Interface:
    1. Creating Posts
    2. Like, Comment, Share Posts
    3. Report System
    4. Category Subscription
    5. Settings
    6. Account Personalization

2.3 User Classes and Characteristics

The main user classes that are anticipated to use this product are young people, gamers, and people interested only in some specific categories we supply. Both the young people and gamer user classes are anticipated to use the service on a high frequency. The latter class is more prone to use the full extent of the gamification side than the first one, given the familiarity with the function. For the class containing people interested only in specific categories, we predict a low use frequency and a low possibility of entering communities other than the ones they are already part of. All classes are offered the same security and privilege levels because in our eyes all users are equal.

2.4 Operating Environment

The operating environment in which the software will operate is any given web browser app that supports HTML 5. The hardware platform, operating system, and other requirements are given by the specific browser used.

2.5 Design and Implementation Constraints

The main limitations are given by the early stages of the project because of the lack of user data we are able to collect. This impedes the correctness and speed of our AI.

2.6 User Documentation

For the average user, there is no need for documentation given the easy-to-use interface and the familiarity given by other social apps. For developers and site owners looking to use our gamification API, there will be a thorough documentation released with the final product.

2.7 Assumptions and Dependencies

The main third-party services used by us come with some risk factors that we assume. One of them is relying on a server hosting service and any problems with it are out of our control. Another one is the storage space of the database which right now is not a problem, but risks rise with the number of users.

3. External Interface Requirements

3.1 User Interfaces

The login interface welcomes the user and lets him log in. Register interface redirects the user to the selection menu so that data about the user's preferences may be collected. After the user's preferences are specified, the main page loads and the feed is given to the user. Interactions and engagement are used so that several badges are unlocked. Several settings are available to the user directly from the navigation bar available on the main page. A settings menu is available for advanced settings that personalise the user's experience. User interaction is made available via hovering over specific posts and consists of 3 types of reactions: like, comment and share. Comments that do not respect the community rules should be reported using the report button available for every post. Login and register systems interact with the user's database and have specific interfaces. So do the feed menu and the settings menu.

3.2 Hardware Interfaces

The supported device types consist of all devices that support a browser, including those that view the website in mobile mode. The used communication protocol is HTTPS and the data is transferred via JSON format.

3.3 Software Interfaces

The website will have an integrated commercial component so that the activity of the users will be monetised. SQLite will be used as an SQL database engine and the interactions with the database will be possible via a REST API. API will receive specific requests in JSON format, including register, login, get the time the user has spent on the website etc.

3.4 Communications Interfaces

HTTPS will be used as a communication standard. Register function also requires e-mail confirmation. Shareable data can be transmitted via a link on other social media apps. As mentioned above, messages between server and client are manipulated in JSON format. Transfer rate depends on the hosting server and internet speed of the user and it is optimised by using webp format for images and imposing a maximum resolution allowed for image posting actions.

4. System Features

4.1 Personalised Feeds

4.1.1 Description and Priority

Every user will get a personalized feed based on the categories they are subscribed to and the posts the AI thinks they will like. This feature is a high priority one, being the main concept of a social app.

4.1.2 Stimulus/Response Sequences

For every user, we have a list of the categories he is currently subscribed to. Using it, we request posts from the database to show on the feed. The AI collects info like the posts liked and commented by the user to calculate what posts or categories to recommend.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

There are no performance requirements for our website, it is designed to work properly on every browser and all platforms.

5.2 Safety Requirements

Being a forum/blog type website where any user can post anything we will have terms and conditions that every user should agree on when will register. Also, every user should respect our rules about posting on the website. If a post won't respect the rules, will be taken down and the user will be banned/muted/timed out from the website. There is a report button to be used if a post is offensive, racist, or contains nudity or explicit context. Everyone should be aware that is not ok to post personal information of any type on the internet, never share personal information with strangers and never trust them.

5.3 Security Requirements

Our website is secure by using HTTPS protocol.

5.4 Software Quality Attributes

GamS is a multi-page website, every page having its HTML, CSS and JavaScript files that makes the cod easy to understand, open for improvment and expansion, and it.s very easy to navigate on it as a customer. We focus more on ease of learning but tried to focus on ease of use as well because both are important: learning new technologies for creating a website but also learning how to create an attractive and fully user interface and a great user experience on our website.

5.5 Business Rules

We identified 3 roles on our website:

  1. The visitor: a user that can see what is on our website, post, and comments, but he can't post, like/share a post, or leave a comment.
  2. The registered user: same as 1) but he can post, like/share a post, or comment on any post.
  3. The admin/moderator: same as 2) but he also can delete posts, ban/mute/timeout users. He is responsible to check reports and decide if the post that has been reported needs to be deleted or not.
